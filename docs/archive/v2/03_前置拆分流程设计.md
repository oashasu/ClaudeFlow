# 前置拆分流程设计（V2追加）

> **日期**: 2026-04-19
> **版本**: V2.03
> **问题来源**: 长任务上下文膨胀，执行质量下降

---

## 问题分析

### 长任务的困境

```
┌─────────────────────────────────────────────────────────────┐
│ 上下文膨胀链                                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  任务："设计Hermes核心模块"                                   │
│      ↓                                                      │
│  Claude Code执行30分钟                                       │
│      ↓                                                      │
│  上下文达到100K tokens                                       │
│      ↓                                                      │
│  Claude压缩历史内容                                          │
│      ↓                                                      │
│  早期决策被遗忘                                              │
│      ↓                                                      │
│  新决策与旧决策冲突                                          │
│      ↓                                                      │
│  执行质量下降                                                │
│      ↓                                                      │
│  需要重做部分工作                                            │
│      ↓                                                      │
│  任务更长...                                                 │
│                                                             │
│  恶性循环                                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 被动触发的问题

```
尝试过的方案：监控上下文 → 提示总结

问题1：监控不准确
• 工具调用计数偏差大（20倍）
• 文件大小≠上下文大小

问题2：提示可能被忽略
• system消息在上下文大时可能被忽略
• Claude可能贪婪继续执行

问题3：Hermes上下文也膨胀
• Hermes需要思考"是否需要拆分"
• Hermes也需要理解任务内容
• 管理器本身也面临上下文膨胀
```

## 方案设计

### 核心思路：前置拆分 + 按计划执行

```
┌─────────────────────────────────────────────────────────────┐
│  关键洞察                                                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  问题根源：                                                  │
│  • 不是"执行过程中需要拆分"                                  │
│  • 而是"任务本身太大，应该拆分后再执行"                        │
│                                                             │
│  解决思路：                                                  │
│  • 拆分在前置阶段完成（Phase 0）                              │
│  • 执行阶段只按计划执行（不做动态决策）                        │
│  • 每个子任务独立执行，上下文可控                              │
│                                                             │
│  类比：                                                       │
│  • 盖楼：先设计楼层，再逐层建造                               │
│  • 不是建造过程中决定"要不要拆层"                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 协调成本控制

### 降低协调成本的三个原则

```
┌─────────────────────────────────────────────────────────────┐
│  协调成本来自"来回沟通"，不是"参与角色多"                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  高成本场景（来回协商）：                                     │
│  • "我觉得应该这样拆，你觉得呢？"                             │
│  • "这个拆分不太合理，能否调整？"                             │
│  • "调整后的方案你同意吗？"                                  │
│                                                             │
│  低成本场景（流水线交付）：                                   │
│  • 拆分Agent输出JSON                                         │
│  • 审查Agent输出adjustments                                  │
│  • Hermes应用adjustments，继续                               │
│                                                             │
│  类比：                                                       │
│  • 来回协商 = 开会讨论（成本高）                              │
│  • 流水线交付 = Git PR review（成本低）                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 产出标准化

```
拆分方案必须JSON格式，固定字段：
→ 拆分Agent和审查Agent不需要"解释"
→ Hermes直接解析，不需要理解
→ 无需来回沟通
```

### 流程固定

```
拆分 →审查 → 确认 → 执行
→ 不是动态协商，而是流水线
→ 每个Agent完成后交给下一个
→ 不需要来回沟通
```

## 实现要点

### 拆分Agent

```python
# hermes/core/task_splitter.py

class TaskSplitter:
    """前置拆分Agent（Sonnet）"""
    
    async def split(self, task_description):
        prompt = f"""
        任务描述：{task_description}
        
        请将此任务拆分成子任务，输出JSON格式：
        {{
          "subtasks": [
            {{ "id": "sub_1", "name": "...", "complexity": 1-3 }}
          ],
          "dependencies": [[ "sub_1", "sub_2" ]],
          "estimated_context_per_subtask": "10K-20K tokens"
        }}
        
        拆分原则：
        • 每个子任务预计上下文不超过20K tokens
        • 复杂度1=简单，2=中等，3=复杂
        • 标注依赖关系
        """
        
        # 调用Sonnet API
        response = await call_sonnet(prompt)
        return json.loads(response)
```

### 审查Agent

```python
# hermes/core/split_reviewer.py

class SplitReviewer:
    """拆分审查Agent（Opus）"""
    
    async def review(self, split_result):
        prompt = f"""
        拆分方案：
        {json.dumps(split_result, indent=2)}
        
        请审查此拆分方案：
        1. 粒度是否合适（每个子任务不超过20K tokens）
        2. 依赖关系是否正确
        3. 是否需要调整
        
        输出JSON格式：
        {{
          "approved": true/false,
          "adjustments": [ "调整建议..." ]
        }}
        """
        
        # 调用Opus API
        response = await call_opus(prompt)
        return json.loads(response)
```

### 执行计划生成

```python
# hermes/core/execution_planner.py

class ExecutionPlanner:
    """生成执行计划"""
    
    def create_plan(self, split_result, review_result):
        # 应用审查调整
        if review_result['adjustments']:
            split_result = self.apply_adjustments(split_result, review_result)
        
        # 生成执行计划
        plan = {
            "subtasks": split_result['subtasks'],
            "order": self.compute_execution_order(split_result['dependencies']),
            "checkpoint_points": [f"sub_{i}_complete" for i in range(len(split_result['subtasks']))]
        }
        return plan
```

## 补充设计：子任务依赖排序算法

### 拓扑排序

```python
# hermes/core/execution_planner.py

def compute_execution_order(self, dependencies):
    """根据依赖关系计算执行顺序（拓扑排序）"""

    # 依赖关系格式：[["sub_1", "sub_2"], ["sub_1", "sub_3"]]
    # 表示sub_2和sub_3依赖sub_1

    # 构建依赖图
    graph = {}
    in_degree = {}

    for dep in dependencies:
        pre, post = dep[0], dep[1]
        graph.setdefault(pre, []).append(post)
        in_degree[post] = in_degree.get(post, 0) + 1
        in_degree.setdefault(pre, 0)

    # Kahn算法拓扑排序
    queue = [k for k, v in in_degree.items() if v == 0]
    order = []

    while queue:
        # 可并行执行的子任务（同一层）
        parallel = queue[:]
        order.append(parallel)

        for node in parallel:
            queue.remove(node)
            for next_node in graph.get(node, []):
                in_degree[next_node] -= 1
                if in_degree[next_node] == 0:
                    queue.append(next_node)

    return order  # [[sub_1], [sub_2, sub_3], [sub_4]]
```

### 并行执行

```
执行顺序：[[sub_1], [sub_2, sub_3], [sub_4]]

Layer 0: sub_1（无依赖）
Layer 1: sub_2, sub_3（并行执行，都依赖sub_1）
Layer 2: sub_4（依赖sub_2和sub_3）

→ 同Layer子任务可分配给不同Worker并行执行
→ 降低总执行时间
```

## 补充设计：复杂度量化标准

### Complexity评分规则

| 级度 | 定义 | 预计工具调用 | 预计上下文 |
|------|------|--------------|------------|
| 1（简单） | 单文件修改，无复杂逻辑 | <50次 | <10K tokens |
| 2（中等） | 多文件修改，需跨文件理解 | 50-100次 | 10K-20K tokens |
| 3（复杂） | 架构级修改，需全局理解 | >100次 | >20K tokens（需拆分） |

### 拆分Agent使用规则

```python
# 拆分时的复杂度判断

prompt = """
拆分原则：
• 每个子任务complexity不超过2（中等）
• 如果预估complexity=3，必须进一步拆分
• complexity=1可合并相邻子任务（减少checkpoint开销）
• 复杂度评估依据：文件数量、逻辑复杂度、依赖数量
"""
```

## 补充设计：拆分拒绝处理流程

### 拒绝场景

```json
// 审查Agent输出
{
  "approved": false,
  "reject_reason": "子任务粒度过粗，sub_7复杂度预估为3",
  "suggestions": [
    "将sub_7拆分为调度核心+队列管理",
    "增加checkpoint频率"
  ]
}
```

### 处理流程

```
审查拒绝 → Hermes读取suggestions → 重新提交拆分Agent（带上suggestions）
→ 拆分Agent输出调整后的方案 → 再次提交审查Agent
→ 最多重试2次

2次后仍拒绝 → 告警 → 用户选择：
1. 强制接受（手动调整）
2. 手动拆分
3. 取消任务
```

### 重试限制

```python
MAX_SPLIT_RETRY = 2  #最多重试2次

# 防止无限来回协商
# 2次后强制进入用户决策
```
