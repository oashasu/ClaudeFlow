# 通信层设计（V2追加）

> **日期**: 2026-04-19
> **版本**: V2.01
> **问题来源**: HTTP REST在人工介入场景实时性不足

---

## 问题分析

### HTTP REST的局限性

| 场景 | HTTP表现 | 问题 |
|------|----------|------|
| 创建任务 | Java调Python | 需要Python暴露HTTP端点 |
| 状态变化 | Python调Java | 可接受，低频 |
| 会话内容 | Python持续调用 | 大体积，高频，阻塞 |
| 人工介入通知 | Python调Java | 可接受 |
| 用户输入传递 | Java调Python | **关键问题**：Python没有HTTP服务监听 |

### 人工介入时序问题

```
T0: Python到达人工介入点
T1: Python调用Java API通知
T2: Java收到，Vue显示弹窗
T3: 用户输入补充信息
T4: Vue提交到Java
T5: Java需要传给Python
    → 问题：Python没有HTTP端点监听
    → 解决方案：轮询？延迟5-30秒
    → 用户期望：立即继续
```

## 方案选择

### 对比矩阵

| 维度 | HTTP双向 | WebSocket | Redis |
|------|----------|-----------|-------|
| 实时性 | 有延迟 | 实时 | 实时 |
| 双向依赖 | 强耦合 | 单向连接 | 单向依赖 |
| Python复杂度 | 需HTTP服务端点 | 仅客户端连接 | redis-py |
| Java复杂度 | Spring Web | Spring WebSocket | Spring Data Redis |
| 额外依赖 | 无 | 无 | Redis |

### 最终选择：WebSocket

**理由**：
- Mac mini单实例，无Redis
- WebSocket无额外依赖
- 实时双向通信
- Python只需客户端连接，无需HTTP服务端点

### Python → Java（WebSocket）

| 消息类型 | 说明 | 频率 |
|----------|------|------|
| `worker_register` | Worker注册身份 | 任务开始时一次 |

| `config_update` | 配置更新 | 管理员操作后 |

| 消息类型 | 说明 | 频率 |

| 端点 | 说明 | 触发 |
|------|------|------|
| `/api/task/pause` | 暂停任务 | 用户点击 |

## 连接管理

### Python Worker生命周期

```
启动：
  asyncio.create_task(websocket.connect(uri))
  await websocket.send({"type":"worker_register", ...})

执行：
  await websocket.send({"type":"tool_call_summary", ...})
  定期心跳（30秒）

关闭：
  await websocket.send({"type":"worker_shutdown", ...})
  await websocket.close()
```

### Java连接状态管理

```java
// WorkerSessionRegistry
Map<String, WebSocketSession> workers;

// 心跳检测
@Scheduled(fixedRate = 30000)
void checkHeartbeat() {
    workers.forEach((id, session) -> {
        if (session.isOpen()) {
            session.sendMessage("{\"type\":\"ping\"}");
        } else {
            workers.remove(id);
        }
    });
}
```

## 人工介入完整时序

```
T0: Python失败3次
    ↓ WebSocket发送 intervention_request（~10ms）
T1: Java收到，存DB，推送SSE给Vue
    ↓
T2: Vue显示弹窗
    ↓ 用户输入
T3: Vue POST /api/intervention/respond
    ↓
T4: Java收到，WebSocket发送 intervention_response（~10ms）
    ↓
T5: Python收到，继续执行

总延迟：用户操作时间 + ~20ms
       （vs HTTP轮询：用户操作时间 + 5-30秒）
```

## 技术实现要点

### Python端

```python
# hermes/core/websocket_client.py
import websockets
import asyncio

class WebSocketClient:
    def __init__(self, uri="ws://localhost:8080/ws/hermes"):
        self.uri = uri
        self.ws = None
    
    async def connect(self, worker_id, task_id):
        self.ws = await websockets.connect(self.uri)
        await self.send({
            "type": "worker_register",
            "worker_id": worker_id,
            "task_id": task_id
        })
    
    async def send(self, message):
        await self.ws.send(json.dumps(message))
    
    async def listen(self):
        async for msg in self.ws:
            data = json.loads(msg)
            if data["type"] == "intervention_response":
                yield data  # 传递给Scheduler处理
```

### Java端

## 补充设计：断线重连机制

### 重连策略

```python
# hermes/core/websocket_client.py

class WebSocketClient:
    # 断线重连参数
    RECONNECT_MAX_ATTEMPTS = 5
    RECONNECT_BASE_DELAY = 1  # 秒
    RECONNECT_MAX_DELAY = 30  # 秒

    async def connect_with_retry(self, worker_id, task_id):
        """带重连的连接"""

        attempt = 0
        while attempt < self.RECONNECT_MAX_ATTEMPTS:
            try:
                self.ws = await websockets.connect(self.uri)
                await self.send({
                    "type": "worker_register",
                    "worker_id": worker_id,
                    "task_id": task_id
                })
                return True
            except Exception as e:
                attempt += 1
                delay = min(
                    self.RECONNECT_BASE_DELAY * (2 ** attempt),
                    self.RECONNECT_MAX_DELAY
                )
                await asyncio.sleep(delay)

        return False  # 重连失败
```

### 重连时机

```
连接断开 → 检测 → 指数退避重连（1s→2s→4s→8s→16s）→ 最多5次
5次失败 → 告警 → 等待用户手动恢复或取消任务
```

## 补充设计：消息确认机制

### 关键消息需ACK

| 消息类型 | 需ACK | 原因 |
|----------|-------|------|
| intervention_request | 是 | 人工介入不能丢失 |
| intervention_response | 是 | 用户输入不能丢失 |
| task_command | 是 | 暂停/恢复/取消不能丢失 |
| tool_call_summary | 否 | 进度丢失可接受 |
| session_content | 否 | 下次会再发送 |

### ACK协议

```
发送方：
{
  "type": "intervention_request",
  "msg_id": "msg_001",
  "content": "...",
  "require_ack": true
}

接收方：
{
  "type": "ack",
  "msg_id": "msg_001",
  "timestamp": "..."
}
```

### 未ACK处理

```
发送关键消息 → 等待ACK（超时5秒）→ 未收到则重发（最多3次）
→ 3次失败 → 告警 → 标记消息为"可能丢失"
```

## 补充设计：多Worker场景

### Worker身份绑定

```python
# 一个任务可能有多个Claude Code进程参与

worker_identity = {
    "worker_id": "worker_001",  # Claude Code进程唯一ID
    "task_id": "task_001",      # 任务ID
    "role": "executor|reviewer|splitter",  # 角色
    "session_id": "session_abc123"  # 会话ID
}
```

### Java端Worker注册管理

```java
// WorkerSessionRegistry.java

public class WorkerSessionRegistry {
    // task_id → List<worker_id>
    private Map<String, List<String>> taskWorkers;

    // worker_id → WebSocketSession
    private Map<String, WebSocketSession> workerSessions;

    public void registerWorker(String taskId, String workerId, WebSocketSession session) {
        taskWorkers.computeIfAbsent(taskId, k -> new ArrayList<>()).add(workerId);
        workerSessions.put(workerId, session);
    }

    public WebSocketSession getSessionByRole(String taskId, String role) {
        // 根据角色查找对应的Worker会话
        // executor → 执行任务
        // reviewer → 审查任务
        // splitter → 拆分任务
    }
}
```

### 多Worker协调

```
Phase0拆分：
├── splitter Worker → 拆分任务
├── reviewer Worker → 审查拆分方案
└── splitter Worker → 应用调整

Phase1-8执行：
├── executor Worker_1 → 子任务1
├── executor Worker_2 → 子任务2（并行）
├── executor Worker_3 → 子任务3

Hermes通过role字段区分Worker，定向发送消息
```
