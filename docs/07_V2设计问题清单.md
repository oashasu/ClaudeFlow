## 问题清单

### P1: 模块边界不清晰

**问题描述**：
新增模块数量多（15+），职责存在重叠，难以明确"谁做什么"。

**具体重叠**：

| 模块A | 模块B | 职责重叠点 |
|-------|-------|------------|
| SessionSummarizer | CheckpointManager | 谁负责"提炼会话内容"？谁负责"保存checkpoint"？ |
| PhaseReviewer | NodeSummarizer | 都是"阶段复盘"，区别是什么？ |
| ToolSummary | SessionParser | 工具调用摘要由谁生成？ |
| ContextManager | CheckpointManager | "上下文清空"和"checkpoint保存"谁先谁后？ |

**影响**：
- 代码实现时职责混乱
- 测试边界难以划定
- 维护成本增加

**状态**: ⏳ 待讨论

---

### P2: 数据流向混乱

**问题描述**：
数据有多条处理分支，缺乏统一的数据流向图。

**当前分支**：

```
分支1（实时进度）：
.jsonl → SessionParser → ToolSummary → WebSocket → Vue显示

分支2（告警）：
.jsonl → ThinkingFilter → 死循环检测 → WebSocket → Vue告警

分支3（提炼存储）：
.jsonl → SessionSummarizer → checkpoint → ?

分支4（复盘）：
checkpoint → Haiku异步复盘 → 知识库 → ?

分支5（人工介入）：
Python → WebSocket → Java → SSE → Vue → 用户 → HTTP → Java → WebSocket → Python
```

**缺失**：
- 统一数据流向图
- 各分支的触发时机
- 分支间的依赖关系
- 数据格式转换点

**影响**：
- 实现时遗漏环节
- 调试困难
- 性能瓶颈难定位

**状态**: ⏳ 待讨论

---

### P3: 角色交互复杂

**问题描述**：
多Agent协作流程复杂，缺少失败处理和超时机制。

**Phase0拆分流程**：

```
用户提交任务 → Hermes创建记录 → 拆分Agent(Sonnet)
→ 审查Agent(Opus) → Hermes应用调整 → 用户确认 → 执行

问题：
• 拆分Agent超时怎么办？（30秒？60秒？）
• 拆分Agent输出格式错误怎么办？
• 审查Agent拒绝后重试流程复杂（最多2次）
• 用户确认超时怎么办？
• 拆分Agent崩溃怎么办？
```

**Phase1-8执行流程**：

```
Hermes → executor Worker → 强制checkpoint → 异步复盘(Haiku)

问题：
• 多个executor Worker并行时如何协调？
• 异步复盘失败但不影响主任务，知识库缺失如何补偿？
• Worker崩溃如何检测？
```

**影响**：
- 用户体验差（等待无反馈）
- 系统不稳定（失败无兜底）
- 状态难以恢复

**状态**: ⏳ 待讨论

### P5: 异常处理不全

**问题描述**：
设计文档缺少各环节的异常处理和兜底方案。

**缺失的异常处理**：

| 场景 | 当前设计 | 缺失 |
|------|----------|------|
| 拆分Agent超时 | 无 | 超时阈值、重试策略、兜底方案 |
| 审查Agent拒绝 | 重试2次 | 2次后仍拒绝的用户处理流程 |
| checkpoint写入失败 | 无 | 重试、备份、告警 |
| WebSocket断线 | 重连5次 | 5次后的处理、消息丢失补偿 |
| 关键消息ACK失败 | 重发3次 | 3次后的处理、强制告警 |
| 异步复盘失败 | 无 | 知识库缺失补偿、下次任务处理 |
| Worker崩溃 | 无 | 检测机制、任务恢复 |
| Claude Code死循环 | 告警 | 告警后的自动处理？强制停止？ |

**影响**：
- 系统脆弱，容易崩溃
- 用户需要手动干预
- 数据丢失

**状态**: ⏳ 待讨论

### P8: Hermes存在的必要性

**问题描述**：
当前设计是"Hermes驱动多个Claude Code"，但是否可以用一个Claude Code去控制其他Claude Code，取代Hermes的作用？Hermes在整个流程中是否不可替换？

**两种架构对比**：

```
方案A（当前设计）：外部管理器模式
┌─────────────────────────────────────────────────────────────┐
│  Hermes（Python程序）                                        │
│      ├── 状态管理（state_machine）                           │
│      ├── 任务调度（scheduler）                               │
│      ├── Checkpoint管理                                      │
│      ├── WebSocket通信                                       │
│      └── 多个Claude Code Worker                              │
│                                                              │
│  Hermes是"外部程序"，独立于Claude Code                       │
└─────────────────────────────────────────────────────────────┘

方案B（替代设计）：Claude-Code-Manager模式
┌─────────────────────────────────────────────────────────────┐
│  Manager Claude Code                                         │
│      ├── 充当"指挥官"角色                                    │
│      ├── 通过Agent工具调用其他Claude Code                    │
│      ├── 自身管理状态和调度                                  │
│      ├── 自身做Checkpoint                                    │
│                                                              │
│  无需外部程序，Claude Code自己管理自己                        │
└─────────────────────────────────────────────────────────────┘
```

**Hermes可能的不可替代点**：

| 功能点 | Hermes能做到 | Claude Code能做到 | Hermes必要性 |
|--------|-------------|------------------|-------------|
| 状态持久化 | ✅ 文件/数据库 | ⚠️ 需要工具调用 | ？ |
| WebSocket通信 | ✅ Python库 | ❌ 无网络能力 | **可能必要** |
| Checkpoint强制触发 | ✅ 外部程序控制 | ⚠️ 需要prompt技巧 | ？ |
| 多Worker协调 | ✅ scheduler | ⚠️ 需要Agent编排 | ？ |
| 死循环检测告警 | ✅ 监控程序 | ❌ 自己无法监控自己 | **可能必要** |
| 用户界面（Vue） | ✅ Java后端 | ❌ 无UI能力 | **必要** |

**需要讨论**：
- Hermes哪些功能是Claude Code无法替代的？
- 如果用Claude-Code-Manager模式，代价是什么？
- Hermes存在的核心价值是什么？

**状态**: ⏳ 待讨论

### P9: 多Agent协同的开源方案调研

**问题描述**：
多Agent协同是通用需求，应该有很多成熟的开源实现方案。大厂是怎么做的？有没有现成的"作业"可以抄？

**已知的多Agent框架**：

| 框架 | 公司/团队 | 核心思路 | 适用场景 |
|------|----------|----------|----------|
| AutoGen | Microsoft | 多Agent对话协作 | 研究、实验 |
| CrewAI | 开源社区 | 角色扮演+任务分配 | 内容创作、业务流程 |
| LangGraph | LangChain | 状态图+节点流转 | 复杂工作流 |
| AgentOps | 商业公司 | Agent监控+调试 | 生产环境运维 |
| Swarm | OpenAI | 轻量级Agent编排 | 简单协作 |
| MetaGPT | 开源社区 | 软件开发流程模拟 | 需求→代码自动化 |
| ... | ... | ... | ... |

**需要调研的问题**：

```
1. 这些框架的核心架构是什么？
   • 是否有"外部管理器"？（类似Hermes）
   • 还是Agent自组织？（类似Claude-Code-Manager）

2. 它们如何解决我们遇到的问题？
   • 状态管理？
   • Checkpoint？
   • 上下文膨胀？
   • 死循环检测？
   • 多Agent协调？

3. 它们的技术选型是什么？
   • 通信方式？
   • 持久化？
   • 监控？

4. 是否可以直接借鉴或集成？
   • 用现有框架替代Hermes？
   • Hermes借鉴框架设计？
```

**调研方向**：
- AutoGen（Microsoft）- 学术领先
- LangGraph（LangChain）- 工作流设计
- CrewAI - 角色分工
- MetaGPT - 软件开发场景
